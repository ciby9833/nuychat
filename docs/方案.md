## Skill Package 标准方案

### 1. 目标

系统中的 AI 能力必须按 **Skill Package** 方式维护，而不是按 connector / executor / tool schema 方式维护。

数据库语义下，每个能力都应支持以下内容：

- `Metadata`
- `SKILL.md`
- `FORMS.md`
- `REFERENCE.md`
- `Scripts`
- `Skill Env`

### 2. 维护形态

虽然系统不是文件系统目录，但编辑与新增必须等价于下面这个结构：

```text
skill/
├── SKILL.md
├── FORMS.md
├── REFERENCE.md
└── scripts/
    └── action.py
```

其中：

- `SKILL.md`
  - 主说明
- `FORMS.md`
  - 参数与表单填写说明
- `REFERENCE.md`
  - 详细接口文档、字段解释、错误码、扫描类型说明
- `scripts/*`
  - 真正执行的脚本
- `skill env`
  - 由这个技能自己维护的环境变量键值

### 3. 角色分工

- 模型
  - 选择 capability
  - 判断是否缺输入
  - 组织自然语言回复

- Skill Package
  - 提供能力说明
  - 提供参数说明
  - 提供参考资料
  - 提供动作脚本

- Script Runtime
  - 真正执行脚本
  - 注入环境变量
  - 返回结构化结果

### 4. 运行方式

```text
用户消息
  -> planner 选 capability
  -> 如果缺参则进入 capability state 澄清
  -> 参数齐全后执行 capability script
  -> 脚本返回 customerReply / response
  -> AI 回复客户
```

### 5. 禁止事项

禁止继续引入或暴露下面这些旧模型：

- connector
- connector version
- connector secret binding
- executor catalog
- builtin executor 作为租户能力模型

这些最多只允许作为内部实现细节，不允许再作为租户配置心智存在。

### 6. 界面要求

前端新增/编辑能力时，只需要少量核心录入框：

- Metadata
- SKILL.md
- FORMS.md
- REFERENCE.md
- Scripts

不再暴露 connector 字段、签名器字段、executor 字段。

### 7. 安全要求

脚本中的密钥、账号、URL 等安全信息，不写死在脚本源码里，而是通过该技能自己维护的环境变量注入：

- `JT_CARGO_API_ACCOUNT`
- `JT_CARGO_PRIVATE_KEY`
- `JT_CARGO_TRACE_URL`

能力里维护 `envBindings`（环境变量键和值）。
这些环境变量属于技能本身，不属于系统全局 `.env`。

### 8. 判定标准

一个能力是否算真正完成，必须满足：

- [x] 能被 planner 选中
- [x] 能在缺参时正常澄清
- [x] 能执行脚本
- [x] 能读取技能内维护的环境变量
- [x] 能返回结构化结果
- [x] 能生成客户可见回复

如果缺少脚本执行层，就不算真正完成。
