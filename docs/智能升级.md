# 智能升级方案

## 1. 目标

当前系统已经不是简单的“聊天机器人 + 几个工具”。

现有基础已经具备：

- `Capability Registry`
- `SuggestCapabilities`
- `Guard`
- `Script Runtime`
- `Tool Result 回填`
- `Customer Intelligence`
- `Copilot`
- `AI / 人工 / Web / WhatsApp / Webhook` 多入口

但系统目前仍然容易出现这类问题：

- AI 能理解问题，但不一定稳定使用最合适的能力
- 多模态输入进入了系统，但未必被最终回复真正消费
- Skill 跑出了结果，但结果不一定被当成“权威事实”使用
- AI 会回答，但缺少独立验证与自动修正，容易“聪明但翻车”
- 人工与 AI 两条链并行存在，但不是完全共享一套智能控制面

本次升级目标不是扩展更多功能，而是把现有系统升级成：

**懂用户、会多模态、能自我纠错、可无缝协同人工的智能客服系统。**

最终实现：

**客户感受不到 AI，却处处被 AI 温暖。**

## 2. 当前实际架构

基于当前代码，系统主链可以分成两条：

### 2.1 AI 座席对客主链

主入口：

- [orchestrator.service.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/orchestrator/orchestrator.service.ts)

当前链路：

1. 读取最近消息与附件
2. 构建 `chatHistory + llmMessages`
3. 加载 `ai_seat` 模型配置
4. 读取租户可用 capabilities
5. 读取用户画像、短期记忆、长期记忆、最近 skill 结果
6. `suggestCapabilities()`
7. `validateCapabilitySuggestions()`
8. 暴露候选原子 tools
9. 最多 3 轮 agent loop
10. 脚本执行
11. tool result 回填给 LLM
12. 输出最终回复 / handoff / clarify
13. 写 working memory、insight、archive、capability state

### 2.2 人工座席辅助主链

主要入口：

- [copilot.service.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/copilot/copilot.service.ts)
- [conversation.routes.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/conversation/conversation.routes.ts)

当前链路分成两种：

1. `Copilot`
   - 读取最近消息
   - 读取用户画像与记忆
   - 读取最近技能结果
   - 用 `agent_assist` 模型生成摘要、意图、建议回复

2. `skills/assist`
   - 读取人工可用 capabilities
   - `suggestCapabilities()`
   - 抽取参数
   - 执行脚本
   - 用 `agent_assist` 模型整理脚本结果，返回前端技能卡片

## 4. 当前系统的关键优点

这些部分必须保留：

### 4.1 Capability Registry 已成立

系统已经从旧 skill 名字硬编码，升级到租户能力目录。

这是正确方向。

### 4.2 Planner 已经弱化

系统已经从“强行选一个技能”开始转向：

- `suggestCapabilities()`

这比旧的单技能规划更合理。

### 4.3 Guard 是独立层

系统已经有：

- 能力候选校验
- tool 调用校验
- runtime governance
- policy 护栏

这是 Harness 的关键基础。

### 4.4 Script Runtime 已成立

系统已经不是 prompt 伪调用，而是：

- capability script
- `stdin` 输入
- `stdout` 结构化输出
- 技能环境变量独立维护

这是系统从“会说”升级到“会做”的关键。

### 4.5 Tool Result 回填已经打通

工具结果不是直接丢给客户，而是先回填给模型，再由模型组织最终回复。

这使得系统具备：

- 能查
- 能理解
- 能综合回答

## 5. 当前系统的核心问题

问题不是功能少，而是**智能控制层不完整**。

### 5.1 缺独立 Verifier / Evaluator

当前链路大多是：

1. 选能力
2. 调工具
3. 工具结果回填
4. LLM 直接输出

缺少独立评估层来判断：

- 结果是否足够支撑回复
- 是否忽略了图片
- 是否与历史 verified facts 冲突
- 是否应该继续调用更多工具
- 是否应该转人工

这是当前“聪明但容易翻车”的第一根因。

### 5.2 缺独立 Reviser / Corrector

当前系统有：

- clarify
- handoff
- guard block

但没有形成真正的自动修正层：

- 工具结果不完整 -> 自动补调下一个工具
- 回答与事实冲突 -> 自动重答
- 图片被忽略 -> 自动重走多模态判断

这导致系统一旦偏航，往往只能：

- 沉默
- 乱答
- 转人工

而不是先自救。

### 5.3 Verified Facts 没有完全独立成权威层

现在系统里有：

- skill result
- memory summary
- profile summary
- insight summary

但还缺一条明确的优先级规则：

**最新已验证工具结果 > 用户画像/记忆总结 > 模型推断**

如果这层不明确：

- 旧推断会盖过新结果
- AI 会把推断当事实
- Copilot 和 AI 座席可能给出不一致建议

### 5.4 多模态是“可输入”，但还不是“强决策信号”

现在系统已经能：

- 把图片带进主链
- 支持图片识别 skill

但当前多模态更多只是“可处理”，还没有成为：

- 与文本等权的决策输入
- verifier 的显式检查项

所以经常出现：

- 有图也只按文字理解
- 图文矛盾时没有纠偏

### 5.5 人工与 AI 还不是同一套智能控制面

现在：

- AI 座席有 agent loop
- Copilot 有自己一条理解链
- `skills/assist` 又是一条辅助链

虽然都在用 capability，但它们还没有完全共享：

- verifier
- reviser
- verified facts
- 多模态证据优先级

结果就是：

- AI 看懂的，人工辅助未必看懂
- 人工辅助拿到的技能卡片，不一定和 AI 座席主链同一判断

## 6. 升级方向

本次升级不扩功能，只升级控制层。

### 6.1 升级一：补 Verifier / Evaluator 层

目标：

把“生成”和“评估”解耦。

新增职责：

1. 校验工具结果是否足够支撑本轮回复
2. 校验最终回复是否和最新 verified facts 冲突
3. 校验本轮是否忽略了图片、附件、语音等多模态输入
4. 校验是否应继续调工具，而不是直接回复
5. 校验是否满足转人工条件

建议落点：

- 新模块：`apps/api/src/modules/ai/verifier.service.ts`

接入位置：

- `tool result -> final answer` 之间
- `Copilot 生成建议 -> 输出前` 之间

最小规则集：

- `missing_multimodal_evidence`
- `answer_conflicts_with_verified_facts`
- `insufficient_tool_evidence`
- `should_continue_tool_loop`
- `should_handoff_to_human`

### 6.2 升级二：补 Reviser / Corrector 层

目标：

把“评估失败”自动收敛成下一步动作，而不是直接翻车。

新增职责：

1. 当 verifier 判定证据不足时，自动续一轮 tool loop
2. 当 verifier 判定回答冲突时，强制重答
3. 当 verifier 判定缺输入时，生成明确澄清
4. 当 verifier 判定高风险时，直接转人工

建议落点：

- 新模块：`apps/api/src/modules/ai/reviser.service.ts`

接入位置：

- `verifier -> orchestrator decision`

最小动作集：

- `continue_tools`
- `rewrite_answer`
- `clarify`
- `handoff`

### 6.3 升级三：抽出 Verified Facts 层

目标：

让系统不再被旧总结、旧推断、旧脏记忆带偏。

建议定义：

`Verified Fact` 只来自这些源：

- capability script result
- 工单/订单/物流/CRM 等系统返回
- 人工确认操作

不包括：

- 模型猜测
- 意图推断
- 画像总结

建议新增模块：

- `apps/api/src/modules/ai/verified-facts.service.ts`

职责：

1. 统一接收脚本结果中的事实字段
2. 生成当前会话的权威事实快照
3. 提供给：
   - orchestrator
   - verifier
   - copilot
   - customer-360

优先级规则：

1. 最新 verified facts
2. 当前 capability state
3. working memory
4. 长期画像/记忆
5. 模型推断

### 6.4 升级四：把多模态从“附件”升级成“证据”

目标：

不是能看图就够了，而是图要真正参与决策。

建议：

1. 在 capability suggester 中，把附件类型作为候选信号
2. 在 verifier 中检查：
   - 客户发了图片，但最终答复完全没提图
3. 在 verified facts 中记录：
   - 图像识别得到的结构化事实

最小升级点：

- 不新增新功能
- 只提升“图文一起进入同一决策面”

### 6.5 升级五：统一 AI 座席与人工座席的智能控制面

目标：

两条链共享同一套智能内核。

统一范围：

- capability registry
- verified facts
- verifier
- reviser
- 多模态输入规则
- 人工介入后的事实回写

这样可以保证：

- AI 主链查到的结果，Copilot 也视为权威
- 人工修正过的内容，AI 后续能学习到
- AI 与人工不再像两套并行系统

## 7. 具体链路调整

### 7.1 AI 座席链路升级

升级后：

1. 读取上下文、多模态、画像、记忆、能力目录
2. `suggestCapabilities()`
3. 候选 tools 进入 agent loop
4. 调 tool
5. 结果进入 `verified facts`
6. verifier 评估：
   - 是否够答
   - 是否要继续调 tool
   - 是否与多模态冲突
7. reviser 决策：
   - 继续工具
   - 重写回答
   - 澄清
   - 转人工
8. 输出最终回复

### 7.2 人工座席辅助链路升级

升级后：

1. Copilot 不再只看文本历史
2. Copilot 读取：
   - 用户画像
   - verified facts
   - 最近技能结果
   - 当前多模态证据
3. skills/assist 返回的不是“原始脚本结果”，而是：
   - 结构化结果
   - verified facts 摘要
   - 下一步建议
4. 人工操作结果反向写回 verified facts / memory

### 7.3 知识库链路升级

目标：

让知识库成为主链中的“工具证据”，不是旁路模块。

调整建议：

1. 知识库查询通过 capability/script 输出结构化结果
2. 结果写入 verified facts
3. verifier 判定：
   - 本轮是否应先查知识库再回复

这样知识库才能稳定参与：

- AI 主链
- Copilot
- 人工辅助

## 8. 功能层需要补足但不扩功能范围的点

这里只列“必须补”，不列新花样功能。

### 8.1 必补：Verifier 结果可观测

需要在日志中明确记录：

- verifier 规则命中
- 为什么继续 tool
- 为什么重答
- 为什么 handoff

### 8.2 必补：Reviser 决策可追踪

要能看到：

- 这次是模型第一次答
- 还是 verifier 打回后重答

### 8.3 必补：Skill 结果结构统一

所有脚本应收敛成统一输出：

- `status`
- `structured`
- `customerReply`
- `response`
- `facts`
- `actions`

避免不同脚本乱返回。

### 8.4 必补：人工确认结果回写

人工做了这些事后，应能回写智能层：

- 确认答案
- 修改答案
- 选择转人工/关闭
- 补充客户事实

否则 AI 无法真正和人工无缝协同。

## 9. 分阶段实施建议

### 阶段 A：补 Verifier

范围：

- AI 座席主链
- Copilot

完成标准：

- 最终回复前有独立评估
- 不再直接“工具结果 -> 最终答复”

### 阶段 B：补 Reviser

范围：

- 只做最小修正动作

完成标准：

- 证据不足能继续 tool
- 回答冲突能自动重答
- 缺输入能稳定澄清

### 阶段 C：建立 Verified Facts

范围：

- capability result
- 人工确认结果
- 工单/订单/物流事实

完成标准：

- AI 与人工都优先读 verified facts

### 阶段 D：统一人工与 AI 控制面

范围：

- Copilot
- `skills/assist`
- AI 座席

完成标准：

- 三者共享同一事实层与评估层

## 10. 最终状态定义

升级完成后的系统，不再是：

- 会聊天
- 偶尔会调工具
- 有时很聪明，有时突然翻车

而是：

- **懂用户**
  - 画像、记忆、历史能真正进入决策
- **会多模态**
  - 文字、图片、附件不是旁路，而是等权证据
- **能自我纠错**
  - 先验证，再修正，再输出
- **可无缝协同人工**
  - AI 与人工共用事实层、建议层、控制层

最终才能实现：

**客户感受不到 AI，却处处被 AI 温暖。**

---

## 11. 本次结论

当前系统不是推翻重来，而是：

- 主骨架已经对
- 缺少 Harness 的关键中间层

最值得做的不是继续堆工具、堆 prompt、堆规则，而是：

1. 补 `Verifier`
2. 补 `Reviser`
3. 抽 `Verified Facts`
4. 统一 AI 与人工智能控制面

这四件事做完，系统才会从“会说、会查”升级到“稳、准、暖”。
