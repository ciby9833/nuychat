# 智能升级方案

## 1. 目标

当前系统已经不是简单的"聊天机器人 + 几个工具"。

现有基础已经具备：

- `Capability Registry`（租户能力目录 + 可用性规则）
- `SuggestCapabilities`（LLM 驱动的候选规划）
- `Guard` 层（planner-guard + runtime-governance + pre-reply-policy + 执行门控）
- `Script Runtime`（真正的子进程 stdin/stdout 能力执行）
- `Tool Result 回填`（工具结果先回填 LLM，再由 LLM 组织回复）
- `Customer Intelligence`（短期记忆 + 长期记忆 + 向量召回 + 画像自动重建）
- `Copilot`（含事实提取 + 优先级排序）
- `AI / 人工 / Web / WhatsApp / Webhook` 多入口
- `Case → Task` 体系（会话 → 案例 → 手工任务 + AI 异步任务）

但系统目前仍然容易出现这类问题：

- AI 能理解问题，但不一定稳定使用最合适的能力
- 多模态输入进入了系统，但未必被最终回复真正消费
- Skill 跑出了结果，但结果不一定被当成"权威事实"使用
- AI 会回答，但缺少**对最终回复质量的独立评估**，容易"聪明但翻车"
- 人工与 AI 共享了 capability 目录，但尚未共享同一套事实源

本次升级目标不是扩展更多功能，而是把现有系统升级成：

**懂用户、会多模态、能自我纠错、可无缝协同人工的智能客服系统。**

最终实现：

**客户感受不到 AI，却处处被 AI 温暖。**

## 2. 当前实际架构

基于当前代码，系统主链可以分成两条，外加一套 Case/Task 执行体系。

### 2.1 AI 座席对客主链

主入口：

- [orchestrator.service.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/orchestrator/orchestrator.service.ts)

当前完整链路：

1. 读取 `ai_seat` 场景模型配置；如无配置直接返回
2. 加载 AI Agent 角色配置（`tenant_ai_agents` — name/role/personality/prompts）
3. 读取最近 20 条消息 + 构建 chatHistory + llmMessages（含 base64 图片）
4. 加载运行时策略（`getBoundRuntimePolicies` — Redis 缓存）
5. 加载租户可用 capabilities + 可用性规则
6. 检查活跃的 capability state（是否处于 clarifying / selected / running 状态）
7. 评估 pre-reply policy 得到首选技能脚本列表
8. 检查最近 5 分钟内是否已有技能结果（`checkRecentSkillContext` — 避免重复调用）
9. `suggestCapabilities()` — LLM 规划候选技能（如果不在续接或已有结果则跳过）
10. `validateCapabilitySuggestions()` — planner guard：置信度 ≥ 0.35 + 有已启用脚本 + 2 分钟内 ≤ 3 次执行
11. **澄清门**：如果候选技能需要用户补充输入 → 存 capability state(status: "clarifying") → 返回澄清问题 → 不进入 tool loop
12. **3 轮 agent loop**（`MAX_TOOL_ROUNDS = 3`）：
    - 调 LLM（含 system prompt + tools）
    - 对每个 tool call：
      - 去重检查（stringified args vs 最近 5 次执行）
      - 执行门控（`evaluateSkillExecutionGate` — 安装/白名单/限流）
      - 候选校验（`validateToolExecutionAgainstCandidates` — 必须在 planner 批准范围内）
      - 允许 → `runCapabilityScriptExecution()` → tool result 回填 LLM
      - 拒绝 → 记录 blocked → 返回错误 JSON 给 LLM
    - LLM 未调 tool → 跳出循环
    - 调了 tool 但未给文本 → 强制最终 LLM 轮（json_object 格式）
13. `normalizeAIInteractionContract()` — 解析 AI 响应
14. `enforcePreReplyPolicy()` — 执行后策略强制（可能覆盖 action）
15. 状态管理：更新 skill_runs / capability_state / working memory / conversation insight
16. 异步调度：memory encoding + execution archive
17. 记录 token 用量
18. 返回 `OrchestratorResult`（action, response, intent, sentiment, skillsInvoked, skillsBlocked）

### 2.2 人工座席辅助主链

主要入口：

- [copilot.service.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/copilot/copilot.service.ts)
- [conversation.routes.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/conversation/conversation.routes.ts)

当前链路分成两种：

1. `Copilot`（含已有事实提取）
   - 读取最近 15 条消息
   - 从最新 `skill_invocations`（decision="allowed"）提取结构化结果
   - `extractLatestFacts()` — 遍历 details/records/events/tracks/history/items 数组，提取 trackingNumber、status、time、location、description
   - 读取客户画像与记忆（`buildCustomerIntelligenceContext`）
   - **事实优先级已实现**：最新 skill facts 覆盖记忆上下文
   - 用 `agent_assist` 模型生成 `{ summary, intent, sentiment, suggestions, entities }`
   - 系统提示明确："latest verified skill result = ground truth (overrides older history)"

2. `skills/assist`
   - 读取人工可用 capabilities
   - `suggestCapabilities()`
   - 抽取参数
   - 执行脚本
   - 用 `agent_assist` 模型整理脚本结果，返回前端技能卡片

### 2.3 Case → Task 执行体系

系统有两套并行的任务模型：

| 系统 | 表 | 创建者 | 用途 |
|------|------|--------|------|
| **手工任务** | `case_tasks` | 座席 / 工作流 | 人工跟进工作项（如"联系物流""准备文档"） |
| **AI 异步任务** | `async_tasks` | orchestrator / task engine | 技能脚本执行、memory encoding、向量索引等后台工作 |

关键关系：

```
conversation → current_case_id → conversation_cases
                                        ↓
                     ┌──────────────────┴──────────────────┐
                case_tasks (手工)                    skill_runs → skill_tasks
                  ↓                                            ↓
           case_task_comments                        async_tasks (AI 后台)
                                                           ↓
                                                    async_task_artifacts
```

- `case_tasks.source_message_id` — 从哪条消息创建的任务（前端"添加任务"按钮）
- `skill_runs.case_id` / `async_tasks.case_id` — AI 任务也归属到 case
- **两套任务互不感知**：AI orchestrator 不读 case_tasks；Copilot 不读 async_tasks 状态

当前问题：

1. 座席手工创建的任务（case_tasks）不会被 AI 感知 — AI 下一轮对话不知道"座席已经安排了物流跟进"
2. AI 技能执行的结果（async_tasks + skill_invocations）不会自动生成 case_tasks — 座席看不到"AI 已经查了物流"的结构化任务卡
3. 任务状态变更（open → in_progress → done）没有事件审计表 — 旧 `ticket_events` 已随 migration 077 删除，`case_tasks` 没有等价替代

## 3. 当前实际信息源

### 3.1 上下文

来自：

- 会话最近消息（orchestrator 取 20 条，copilot 取 15 条）
- 客户最新附件（base64 编码进 LLM）
- recent skill context（最近 5 分钟内的技能结果）

现状：

- 已支持文本 + 附件共存
- AI 主链可读图片
- 人工辅助链已补齐附件参数传递

结论：

- 这一层基础是对的
- 不需要推翻

### 3.2 用户画像

来自：

- `customer_memory_profiles`（自动重建的综合画像）
- `conversation_memory_snapshots`（会话级快照）
- `customer_memory_states`（客户活跃状态）

代码：

- [customer-intelligence.service.ts](/Users/ellis/Documents/nuychat/nuyess-chat/apps/api/src/modules/memory/customer-intelligence.service.ts)

画像重建机制（`rebuildCustomerProfile`）：

- 聚合最近 10 条快照 + 18 条记忆单元 + 8 条状态
- 过滤 preferences/facts/traits
- 生成 profileSummary（≤6500 字符）+ soulProfile + operatingNotes
- 自动计算 profileKeywords（token ≥ 3 字符，max 64）

现状问题：

- 画像更多是"补充上下文"
- 还不是"强决策输入"
- 画像重建是异步的，新事实写入后不一定立即可用

### 3.3 记忆

包括：

- working memory（Redis，TTL 4h，max 20 turns）
- memory units（`customer_memory_units` — 词法 + 向量混合检索）
- memory states（`customer_memory_states` — 活跃状态快照）
- vector recall（`searchRelevantMemoryVectors` — 向量相似度搜索）

检索打分算法（`buildCustomerIntelligenceContext`）：

- 词法得分 × 0.42 + 向量得分 × 0.42 + salience × 0.1 + 时效衰减 × 0.06
- 30 天指数衰减
- 最多返回 6 条

现状问题：

- 已验证事实与模型推断尚未完全分层
- 旧会话推断可能污染新一轮决策
- 记忆 fingerprint 去重已实现（SHA1），但同一事实的不同表述可能生成不同 fingerprint

### 3.4 技能

来自：

## 4. 当前系统的关键优点

这些部分必须保留：

### 4.1 Capability Registry 已成立

系统已经从旧 skill 名字硬编码，升级到租户能力目录 + 多维可用性规则（channel/role/module/owner_mode）。

这是正确方向。

### 4.2 分布式 Guard 层已成型

系统已有的验证/防护机制：

| 组件 | 位置 | 职责 |
|------|------|------|
| `planner-guard.service.ts` | agent-skills/ | 候选校验（confidence ≥ 0.35）+ 防重复（2min/3次）+ 去重 |
| `runtime-governance.service.ts` | skills/ | 执行门控（安装/白名单/限流/审计） |
| `pre-reply-policy.service.ts` | ai/ | 回复前策略评估 + 强制执行 |
| `conversation-closure-evaluator.service.ts` | conversation/ | 会话关闭评估（verdict + confidence + reason） |
| Copilot 事实优先级 | copilot/ | skill result > memory > 推断 |
| Budget gate | ai/ | token 预算门控 |

这已经是 Harness 的关键基础，**不是从零开始**。

### 4.3 Planner 已经弱化

系统已经从"强行选一个技能"转向：

- `suggestCapabilities()` — LLM 驱动多候选规划
- 最多 5 个候选，按置信度排序
- 与 clarification gate 联动

这比旧的单技能规划更合理。

### 4.4 Script Runtime 已成立

系统已经不是 prompt 伪调用，而是：

- 真实子进程（Python venv / Node / Bash）
- stdin 输入 JSON，stdout 输出结构化
- 环境变量独立维护
- 依赖缓存（Python venv SHA1 缓存）
- 超时控制（默认 15s）

### 4.5 Tool Result 回填已经打通

工具结果不是直接丢给客户，而是先回填给 LLM，再由 LLM 组织最终回复。

### 4.6 DAG 任务图已支持

`skill-task-runtime.service.ts` 实现了完整的 DAG 任务图：

- 节点有依赖关系（`depends_on`）
- 支持级联解锁（上游成功 → 下游 planned → queued）
- 支持级联阻塞（上游失败 → 下游 blocked）
- 这是"多步骤复杂技能编排"的现成基础

### 4.7 Capability State Machine 已成立

`conversation_capability_states` 实现了完整的技能执行状态机：

- `clarifying` — 等用户补充输入
- `selected` — 技能已选定
- `running` — 执行中

含 `missingInputs`、`resolvedInputs`、`clarificationQuestion` 字段，支持多轮澄清。

## 5. 当前系统的核心问题

问题不是功能少，也不是"没有验证层"，而是**现有验证层只覆盖了"执行边界"（能不能调 tool），缺少对"回复质量"的独立评估**。

### 5.1 缺对最终回复质量的独立评估

当前 Guard 层覆盖范围：

- ✅ 能不能调这个 tool（planner-guard + execution-gate + runtime-governance）
- ✅ 预算够不够（budget gate）
- ✅ 策略允不允许（pre-reply-policy）
- ✅ 是否重复调用（dedup guard）

**未覆盖**：

- ❌ 工具结果是否足够支撑本轮回复
- ❌ 最终回复是否与最新已验证事实冲突
- ❌ 本轮是否忽略了图片/附件等多模态输入
- ❌ 是否应该继续调 tool 而不是直接回复
- ❌ 回复是否满足转人工条件

这是当前"聪明但容易翻车"的第一根因。

### 5.2 缺独立修正层

当前系统有：

- clarify（澄清门）
- handoff（转人工）
- guard block（执行拒绝）

但没有形成真正的自动修正层：

- 工具结果不完整 → 应自动补调下一个 tool
- 回答与事实冲突 → 应自动重答
- 图片被忽略 → 应自动重走多模态判断

当前偏航后只能：沉默、乱答、或转人工。不能先自救。

### 5.3 事实提取已有但未公共化

Copilot 的 `extractLatestFacts()` 已经能从 skill_invocations 提取结构化事实，并且 prompt 中明确了"verified skill result = ground truth"。

**但这个能力只在 Copilot 链路中使用**。Orchestrator 的 agent loop 中不使用相同的事实提取和优先级排序逻辑。

需要做的不是"从零建 Verified Facts 层"，而是：

1. 把 `extractLatestFacts()` 抽成公共函数
2. 让 orchestrator 和 copilot 共享同一套事实
3. 明确优先级：最新 verified facts > capability state > working memory > 长期画像 > 模型推断

### 5.4 多模态是"可输入"，但还不是"强决策信号"

系统已经能把图片带进主链、支持图片识别技能。

但多模态更多只是"可处理"，还没有成为：

- 与文本等权的决策输入
- Verifier 的显式检查项

所以经常出现有图也只按文字理解、图文矛盾时没有纠偏。

### 5.5 两套任务体系互不感知

当前 `case_tasks`（手工）和 `async_tasks` + `skill_invocations`（AI）互不读取：

- AI orchestrator 不知道座席手工创建了什么任务
- 座席在 Copilot 面板看不到 AI 执行了什么异步任务
- 手工任务状态变更没有事件审计（旧 `ticket_events` 已删除）
- AI 技能结果不自动生成可见的 case_task 卡片

这导致 AI 与人工像两套并行系统，无法真正协同。

## 6. 升级方向

本次升级不扩功能，只升级控制层和信息层。

### 6.1 升级一：抽取公共 Fact Layer

**优先级最高** — 它是 Verifier 的前提。

目标：

让 orchestrator、copilot、skills/assist 三条链共享同一套事实源和优先级规则。

具体做法：

1. 从 copilot 的 `extractLatestFacts()` 抽出为公共函数
2. 新模块：`apps/api/src/modules/ai/fact-layer.service.ts`
3. 职责：
   - 从 `skill_invocations`（最新 allowed 记录）提取结构化事实
   - 从 `case_tasks`（当前 case 的手工任务）提取任务状态事实
   - 从 `customer_memory_states`（活跃状态）提取状态事实
   - 统一输出 `FactSnapshot`：`{ verifiedFacts[], taskFacts[], stateFacts[], lastUpdated }`
4. 接入位置：
   - orchestrator：每轮 tool result 回填后，刷新 FactSnapshot
   - copilot：生成前读取 FactSnapshot
   - skills/assist：返回结果时写入 FactSnapshot

优先级规则（写入 service 内部，不靠 prompt）：

1. 最新 verified facts（来自脚本执行结果）
2. 当前 capability state（clarifying / running 状态信息）
3. 当前 case tasks 状态（手工任务进度）
4. working memory（Redis 短期记忆）
5. 长期画像 / 记忆
6. 模型推断

`Verified Fact` 只来自这些源：

- capability script result（`skill_invocations.result`）
- 工单/订单/物流/CRM 等系统返回
- 人工确认操作（seat 修改 case_task 状态）

不包括：

- 模型猜测
- 意图推断
- 画像总结

### 6.2 升级二：补 Verifier / Evaluator 层 ✅

目标：

把"生成"和"评估"解耦。在两个关键点插入评估：

**插入点 A：agent loop 内部（每轮 tool result 回填后、下一轮 LLM 调用前）** ✅

评估：
- ✅ 当前证据是否已足够回答客户问题
- ✅ 是否还需要继续调 tool

**插入点 B：最终回复生成后、`enforcePreReplyPolicy()` 之前** ✅

评估：
- ✅ 最终回复是否与 FactSnapshot 中的 verified facts 冲突
- ✅ 本轮是否忽略了多模态输入
- ✅ 是否满足转人工条件

落点（模块化文件结构）：

- ✅ `apps/api/src/modules/ai/verifier/types.ts` — 类型定义（VerifierVerdict, RuleFinding, PointAContext, PointBContext, VerifierRule）
- ✅ `apps/api/src/modules/ai/verifier/verifier.service.ts` — 主服务（evaluatePointA, evaluatePointB, action resolution）
- ✅ `apps/api/src/modules/ai/verifier/rules/insufficient-evidence.ts` — 证据不足规则
- ✅ `apps/api/src/modules/ai/verifier/rules/fact-conflict.ts` — 事实冲突规则
- ✅ `apps/api/src/modules/ai/verifier/rules/missing-multimodal.ts` — 多模态遗漏规则
- ✅ `apps/api/src/modules/ai/verifier/rules/should-continue-tools.ts` — 继续调 tool 规则
- ✅ `apps/api/src/modules/ai/verifier/rules/should-handoff.ts` — 转人工规则
- ✅ `apps/api/src/modules/ai/verifier/index.ts` — 公共 API 导出

最小规则集（5 条全部实现）：

- ✅ `insufficient_tool_evidence` — 证据不足（FactSnapshot 为空但客户问了具体问题）；Point A + B
- ✅ `answer_conflicts_with_verified_facts` — 回复与最新事实冲突（基于规则的轻量检查）；Point B
- ✅ `missing_multimodal_evidence` — 客户发了图但回复完全没提；Point B
- ✅ `should_continue_tool_loop` — 应该继续调 tool（tool error/无 verified facts）；Point A
- ✅ `should_handoff_to_human` — 应该转人工（客户明确要求/情绪极端/连续失败）；Point B

集成：
- ✅ orchestrator.service.ts — Point A 评估结果记录到 verifierSteps
- ✅ orchestrator.service.ts — Point B 评估结果可覆盖 AI decision（handoff override）
- ✅ verifierSteps 记录到 ai_traces（通过 scheduleExecutionArchive）

### 6.3 升级三：补 Reviser / Corrector 层

目标：

把"评估失败"自动收敛成下一步动作，而不是直接翻车。

新增职责：

1. 当 verifier 判定证据不足时 → 自动续一轮 tool loop
2. 当 verifier 判定回答冲突时 → 强制重答（重走最终 LLM 轮）
3. 当 verifier 判定缺输入时 → 生成明确澄清
4. 当 verifier 判定高风险时 → 直接转人工

建议落点：

- 新模块：`apps/api/src/modules/ai/reviser.service.ts`

接入位置：

- 插入点 A 之后：`tool result → verifier → reviser → 决定是继续 tool 还是输出`
- 插入点 B 之后：`最终回复 → verifier → reviser → 决定是重答、澄清还是 handoff`

最小动作集：

- `continue_tools` — 补调下一个 tool（复用现有 agent loop，而非硬编码新调用）
- `rewrite_answer` — 携带 FactSnapshot 重走最终 LLM 轮
- `clarify` — 生成澄清问题 + 存 capability state
- `handoff` — 转人工

**注意**：Reviser 只用于 AI 座席主链。Copilot 链路不需要 Reviser — 人工座席本身就是 Reviser。

### 6.4 升级四：把多模态从"附件"升级成"证据"

目标：

不是能看图就够了，而是图要真正参与决策。

建议：

1. 在 capability suggester 中，把附件类型作为候选信号
2. 在 verifier 中检查：
   - 客户发了图片，但最终答复完全没提图
3. 在 fact layer 中记录：
   - 图像识别得到的结构化事实

最小升级点：

- 不新增新功能
- 只提升"图文一起进入同一决策面"

### 6.5 升级五：统一 AI 与人工的信息源

目标：

两条链共享同一套**信息源**（不是控制层 — Copilot 不需要 Reviser）。

| 共享层 | AI 座席 | Copilot | skills/assist |
|--------|---------|---------|---------------|
| Fact Layer | 读 + 写 | 只读 | 写 |
| capability registry | 使用 | 展示 | 使用 |
| case_tasks 状态 | **新增：只读** | **新增：只读** | — |
| 多模态证据 | 使用 | 展示 | — |

不统一的部分（各链路保留独立控制）：

| 控制层 | AI 座席 | Copilot |
|--------|---------|---------|
| Verifier | 需要 | 不需要（座席自己判断） |
| Reviser | 需要 | 不需要（座席自己修正） |
| Agent loop | 3 轮 | 无 |

这样可以保证：

- AI 主链查到的结果，Copilot 也视为权威
- 座席手工创建的任务，AI 下一轮能感知
- 但不会把 Copilot 搞成另一个 orchestrator

### 6.6 升级六：打通两套任务体系

目标：

让 AI 任务和手工任务在座席工作台上可见为统一视图。

具体做法：

1. AI 技能执行完成后，自动生成一条 `case_task`（状态 done，creator_type: "ai"）— 让座席看到 AI 做了什么
2. 座席创建 case_task 后，将任务摘要写入 FactSnapshot — 让 AI 下一轮能感知
3. case_task 状态变更写审计事件（补回 migration 077 删掉的 ticket_events 等价功能）

## 7. 具体链路调整

### 7.1 AI 座席链路升级

升级后：

1. 读取上下文、多模态、画像、记忆、能力目录
2. **读取 FactSnapshot**（含 verified facts + case_tasks 状态）
3. `suggestCapabilities()`
4. 候选 tools 进入 agent loop
5. 调 tool
6. 结果写入 FactSnapshot
7. **Verifier 评估（插入点 A）**：
   - 证据够不够
   - 要不要继续调 tool
8. **Reviser 决策**：
   - `continue_tools` / `proceed_to_answer`
9. 输出最终回复
10. **Verifier 评估（插入点 B）**：
    - 回复与事实冲突否
    - 多模态证据有没有被忽略
11. **Reviser 决策**：
    - `rewrite_answer` / `clarify` / `handoff` / `accept`
12. enforcePreReplyPolicy
13. 状态管理 + 异步调度

### 7.2 人工座席辅助链路升级

升级后：

1. Copilot 读取 FactSnapshot（公共函数，不再自行从 skill_invocations 提取）
2. Copilot 读取 case_tasks 状态（"座席已安排物流跟进"）
3. Copilot 用统一事实生成建议
4. skills/assist 返回的不是"原始脚本结果"，而是：
   - 结构化结果
   - 写入 FactSnapshot
   - 自动生成 case_task 卡片
5. 座席操作结果反向写回 FactSnapshot

### 7.3 知识库链路升级

目标：

让知识库成为主链中的"工具证据"，不是旁路模块。

调整建议：

1. 知识库查询通过 capability/script 输出结构化结果
2. 结果写入 FactSnapshot
3. Verifier 判定：
   - 本轮是否应先查知识库再回复

这样知识库才能稳定参与 AI 主链、Copilot、人工辅助三条链。

## 8. 功能层需要补足但不扩功能范围的点

这里只列"必须补"，不列新花样功能。

### 8.1 必补：Verifier 结果可观测

需要在 `ai_traces` 中明确记录：

- verifier 规则命中（哪条规则触发了什么动作）
- 为什么继续 tool
- 为什么重答
- 为什么 handoff

扩展 `ai_traces.steps` JSONB：加入 `{ step: "verifier", rules_triggered: [...], decision: "..." }` 节点。

### 8.2 必补：Reviser 决策可追踪

要能看到：

- 这次是模型第一次答
- 还是 verifier 打回后重答
- 重答了几次

同样记录在 `ai_traces.steps` 中。

### 8.3 必补：Skill 结果结构统一

所有脚本应收敛成统一输出：

- `status` — 执行状态
- `structured` — 结构化数据
- `customerReply` — 可直接发给客户的文本
- `response` — 完整响应
- `facts` — 可写入 FactSnapshot 的事实列表
- `actions` — 建议后续动作

避免不同脚本乱返回。

### 8.4 必补：case_task 事件审计

migration 077 删掉了 `ticket_events`，但 `case_tasks` 没有等价的事件审计表。需要补：

- `case_task_events`（append-only）
- event_type: created | status_changed | priority_changed | assigned | commented
- 供 Copilot 和 AI 读取任务变更历史

### 8.5 必补：人工操作结果回写

座席做了这些事后，应能回写 Fact Layer：

- 确认答案
- 修改答案
- 选择转人工/关闭
- 补充客户事实
- 创建/完成 case_task

否则 AI 无法真正和人工无缝协同。

## 11. 最终状态定义

升级完成后的系统，不再是：

- 会聊天
- 偶尔会调工具
- 有时很聪明，有时突然翻车

而是：

- **懂用户**
  - 画像、记忆、历史能真正进入决策
- **会多模态**
  - 文字、图片、附件不是旁路，而是等权证据
- **能自我纠错**
  - 先验证，再修正，再输出
- **可无缝协同人工**
  - AI 与人工共用事实层，任务互通

最终才能实现：

**客户感受不到 AI，却处处被 AI 温暖。**

## 12. 本次结论

当前系统不是推翻重来，更不是"从零补 Harness"。

系统已有完整的分布式 Guard 层（planner-guard + execution-gate + runtime-governance + pre-reply-policy + budget-gate），**缺的不是防护，而是对回复质量的闭环评估**。

最值得做的四件事：

1. **抽取公共 Fact Layer** — 让三条链共享事实源（前提条件）
2. **补 Verifier** — 评估回复质量（核心升级）
3. **补 Reviser** — 自动修正（可靠性保障）
4. **打通任务体系** — AI 与人工任务互通（协同基础）

这四件事做完，系统才会从"会说、会查"升级到"稳、准、暖"。
