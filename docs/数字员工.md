
# 数字员工重构蓝图

**最后修订**：2026-04-13  
**适用范围**：`apps/api/src/modules`（当前 AI 主链、Copilot、知识库、技能编排、路由与记忆系统）

## 1. 结论与核心问题

NuyChat 当前 AI 架构本质上是一个**事务执行型客服 Agent**，而非真正的**数字员工**。

**当前强项**：技能选择、工具调用、事实回填、风控、转人工。  
**当前弱项**：缺乏“先理解业务知识、再决定执行”的高级能力。

**根本原因**：主链顺序为 `skills → tools → answer`，而非 `route → knowledge → selective tools → answer`。

**导致的问题**：
- 非交易类咨询被过早拉入工具轨道
- 知识库存在但未成为对话主链的一等公民
- 物流/订单能力过度占据默认心智
- 护栏过多且集中在“别乱答”，而非“先用知识解决问题”

**重构目标**：将系统升级为真正的数字员工，实现：
- 先判断处理轨道
- 能用知识解决的优先用知识
- 仅在需要动作执行时暴露工具
- 工具结果与知识结果进入统一证据层
- 护栏前移至轨道决策，后置护栏仅保留高价值部分

## 2. 当前架构分析

### 2.1 当前主链流程（orchestrator.service.ts）
1. 加载运行时策略、租户 skills、会话状态
2. 执行 preReplyPolicy 筛 preferred bindings
3. 进行 capability planning
4. 组装 prompt（注入 candidate skills 文档）
5. 暴露 runtime tools 并执行 3 轮 tool loop
6. 执行 verifier / reviser / pre-reply-policy 后置修正

### 2.2 当前最大偏差
- 先加载全量能力目录 → 再挑选 → 再给工具 → 最后补救
- 与数字员工要求的“先定性（轨道判断）→ 先知识 → 再按需工具”完全相反

### 2.3 主要问题汇总
- Orchestrator 单体过于臃肿
- Guard/Verifier/Reviser 层层重复判断（逻辑套娃）
- Candidate skill 文档整包注入 prompt（严重 token 浪费）
- 知识库未接入主回答链
- 多处硬编码（如 triggerHints 写死为空、意图识别依赖正则等）

## 3. 目标架构

### 3.1 设计原则
1. 先定性，后定量
2. 先知识，后工具
3. 工具按需暴露，不提前污染 prompt
4. 护栏前移到轨道决策，后置仅保留高价值部分

### 3.2 目标主链
```text
消息到达
  → Semantic Routing
      → Knowledge Track
      → Action Track
      → Clarification Track
```

### 3.3 三条处理轨道

| 轨道              | 适用场景                           | 核心原则                          | 工具暴露 |
|-------------------|------------------------------------|-----------------------------------|----------|
| **Knowledge Track** | FAQ、业务规则、产品知识、售前咨询、流程说明 | 默认不暴露工具，用知识库优先解决 | 禁用    |
| **Action Track**    | 查订单/物流/账单、退款、取消、建单 | 只装载相关 skills，事实驱动回答   | 按需最小集 |
| **Clarification Track** | 意图不清、缺槽位、模糊请求       | 优先追问最小必要信息             | 禁用    |

## 4. 关键模块蓝图

### 4.1 新增/重点模块

- **Semantic Routing Layer**：`semantic-router.service.ts`
- **Knowledge Context Pipeline**：`knowledge-retrieval.service.ts` + `knowledge-rerank.service.ts` + `knowledge-context-builder.ts`
- **Dynamic Skill Hydration**：`skill-hydration.service.ts`
- **Unified Evidence Layer**（扩展现有 Fact Layer）
- **Answer Composer**：`answer-composer.service.ts`

### 4.2 统一证据层（Unified Evidence Layer）
优先级（由高到低）：
1. 最新 verified tool facts
2. 最新高置信 knowledge facts
3. Active task / state facts
4. Customer memory / profile
5. Older conversation inference

## 7. 代码文件注释执行规范（必须遵守）

所有 AI 主链相关文件**文件头**必须包含以下注释模板：

```ts
/**
 * 作用：...
 * 上游：...
 * 下游：...
 * 协作对象：...
 * 不负责：...
 * 变更注意：...
 */
```

## 8. 核心验收用例

1. 知识问答（产品、流程、政策）→ Knowledge Track，不调工具
2. 动作执行（订单、物流、退款）→ Action Track，正确调用最小工具集
3. 意图模糊/缺信息 → Clarification Track，主动追问最小必要信息
4. 多轮对话切换轨道 → 能正确切换，不被历史 skill 锁定
5. Copilot 与 AI 回复口径一致（统一证据层）

## 9. 回归风险重点关注
- Semantic Route 分类错误
- Knowledge recall 质量不足
- Skill hydration 过窄导致工具缺失
- 精简 Guard 后暴露历史安全问题
- Trace 结构变更影响运营排查

建议：每个 Phase 完成后进行灰度发布 + 完整回归测试。
10. 最终目标效果
重构完成后，NuyChat 数字员工应具备以下表现：

面对知识咨询时像业务专家，而非查单机器人
仅在需要执行动作时进入工具轨道
面对模糊问题时主动、精准追问
Copilot 与 AI 使用同一套证据体系，口径一致
整体链路更短、token 成本更低、可维护性更高
