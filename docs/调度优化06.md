# 调度优化 06

## 1. 目标

本次方案的目标不是继续给现有链路打补丁，而是把当前“入站调度、AI handoff、SLA 超时改派、人工请求排队”收敛成一套统一的决策与执行模型，满足：

1. 租户默认可用，不依赖复杂规则。
2. 用户明确要求人工时，优先人工；无人可服务时，回退 AI。
3. 已进入人工接管态的会话，后续消息不能再被普通路由拉回 AI。
4. SLA 超时后的改派必须复用同一套调度决策，不允许单独再造一条“超时专用调度”。
5. 座席忙碌、排班、在线状态、排队压力的判断必须可解释，且数据来源明确。
6. AI 向客户的提示只负责自然语言表达，决策本身由系统完成，不允许硬编码词表驱动主流程。

### 4.1 当前生效的 4 类时效

1. `first_response`
2. `assignment_accept`
3. `subsequent_response`
4. `follow_up`

### 4.2 每类的真实触发和动作

#### `first_response`

触发入口：

- 入站后 `deriveInboundTimeoutPlan()`

动作：

- 写 `sla_breaches`
- `alert / escalate`
- **不重分配**

#### `assignment_accept`

触发条件：

- 会话在人工队列中
- `queueStatus in assigned/pending`
- 仍未被真正接手

动作：

- 写 `sla_breaches`
- 若策略有 `reassign`
  - 当前已经重新走统一调度 `createPlan`
  - 不再只走人工分配器

#### `subsequent_response`

触发条件：

- 服务方已回复过
- 客户再次发消息
- 当前 owner 未继续回复

动作：

- 写 `sla_breaches`
- 若条件满足 `owner_unavailable / always`
  - 调用 `applySmartReassignment()`

#### `follow_up`

触发条件：

- 服务方已回复
- 进入 follow_up 观察

动作：

- 超时后读取**当前** follow-up 配置
- 若为 `semantic`
  - 调 LLM 关闭评估器
  - 评估基于当前 `case_id` 的摘要 + 最近消息
- 若判断继续
  - 重新挂 follow_up
- 若判断关闭
  - `closeCurrentSegment`
  - `resolve_conversation`

### 4.3 当前 follow_up 语义的结论

当前 `follow_up` 已经比之前正确：

- 不再只看最后一句
- 不再看整条 conversation 全历史
- 现在是“当前 case 上下文 + 有界截断”

但它仍有一个设计问题：

- `follow_up` 同时承担“自然关闭评估”和“轮次结束控制”
- 与“人工请求排队中的继续等待/放弃等待”还没有解耦

## 5. 当前最关键的真实问题

### 问题 1：决策统一，但状态机不统一

当前不是多套完全独立调度器，而是：

- 一个主决策器
- 多个特殊入口
- 多个状态补丁

这比完全分裂要好，但仍然会导致：

1. 人工请求进入队列后，还需要 worker 补状态
2. AI handoff 与普通路由不是同一类决策
3. SLA 改派虽已回统一调度，但仍不是同一状态机

### 问题 2：当前没有“人工请求”的专用状态模型

现在人工请求主要依赖：

- `intent`
- `handoff_required`
- `queue_status`
- worker 中的 handoff 逻辑

缺失的是明确的结构化状态：

- 当前是否处于“用户坚持人工”
- 当前是否允许回退 AI
- 当前是否已经分配给某个具体人工
- 当前是否只是“无人工可服务而等待”

### 问题 3：排队信息没有成为结构化事实

当前系统没有标准字段表达：

- `queuePosition`
- `estimatedWaitSec`
- `assignedHumanBacklog`
- `humanAvailabilitySummary`

因此只能：

- 分配成功 / pending
- 却无法为“人工排队中”的客户生成高质量、可解释的回复

### 问题 4：用户明确请求人工时，回退 AI 的逻辑不完整

目标语义应当是：

1. 有可服务人工 -> 必须优先人工
2. 无可服务人工 -> 回退 AI
3. AI 回复时要带上结构化的人类可服务事实，交给 LLM 自然表达

当前实现只覆盖了部分：

- 已进入 human handoff queue 的会话不会再回普通 AI
- 但“没有任何人工可服务时的受控回退 AI + 自然语言说明”还没有被正式建模

## 6. 目标架构

### 6.1 统一决策入口

保留唯一入口：

- `RoutingDecisionService.createPlan(context, trigger)`

所有触发只以不同 `triggerType` 进入，不再暴露多套 plan 构造器。

建议只保留这类 trigger：

- `inbound_message`
- `ai_handoff`
- `agent_handoff`
- `assignment_accept_timeout`
- `subsequent_response_timeout`
- `manual_transfer`
- `supervisor_transfer`

不再保留独立的：

- `createAgentHandoffPlan`
- `createAiHandoffHumanPlan`

### 6.2 统一输出结果

建议统一成 5 类决策结果：

1. `assign_human_immediately`
2. `assign_human_with_queue_notice`
3. `fallback_to_ai_human_unavailable`
4. `assign_ai_normally`
5. `hold_pending_no_serviceable_owner`

每一种结果都必须结构化返回：

| 字段 | 含义 |
|---|---|
| `selectedOwnerType` | `human / ai` |
| `selectedAgentId / aiAgentId` | 实际目标 |
| `queueMode` | `none / assigned_waiting / pending_unavailable` |
| `queuePosition` | 当前前面等待人数 |
| `estimatedWaitSec` | 预计等待时长 |
| `humanRequestState` | `none / requested / queued / fallback_to_ai` |
| `llmNoticeFacts` | 用于让 LLM 组织自然语言的事实集 |

### 6.3 决策与文案解耦

系统负责：

- 是否转人工
- 是否回退 AI
- 分给谁
- 队列位置是多少
- 预计等待多久
- 当前人工是否在线/上班

LLM 只负责：

- 把这些事实组织成自然语言回复

不能让 LLM 决定：

- 是否排队
- 是否回 AI
- 是否释放人工队列

     - `service_request_mode = ai_opt_in`
     - 统一调度强制优先 AI
     - 不再继续锁定人工侧

4. 排队中客户通知
   - `routing_notice` 会收到 `switchBackCommand = "#AI"` 这一结构化事实
   - LLM 仅负责把这个事实自然语言表达给客户
   - 系统不再依赖“理解客户每句是否还想要人工”来做迁移

本阶段刻意没有做的事：

- 没有引入“排队中消息意图分类器”
- 没有对普通补充信息做语义转向
- 没有增加新的业务路由规则
- 没有让 `service-mode engine` 决定具体分给哪个人工/AI

也就是说：

- `routing-decision` 仍只负责“分给谁”
- `service-mode` 只负责“当前诉求是什么、人工进度到了哪一步、客户是否明确切回 AI”

## 9. 最终结论

当前系统的问题不是“没有统一调度器”，而是：

1. 主决策器之外仍有 handoff / timeout / queue 状态补丁
2. `handoff_required` 承担了过多语义
3. 人工请求没有专用状态模型
4. 排队与 ETA 没有被结构化建模

因此本次改造不建议继续补丁式修正，而应该：

1. 先统一状态模型
2. 再统一决策入口
3. 最后再补排队提示与 AI 回退说明

只有这样，后续“人工请求优先、无人自动回 AI、排队动态提示、SLA 超时统一改派”才能变成一套可解释、可维护、可扩展的系统，而不是继续在 worker 和状态字段上套娃。
