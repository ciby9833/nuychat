# AI 工具技能维护培训文档

本文用于培训租户管理员、运营人员和实施人员如何维护 `AI 工具技能`。  
文档内容以当前系统实际实现为准，对应租户管理端菜单：

- `平台配置 > AI 工具技能`

这份文档重点回答 4 个问题：

1. `AI 工具技能` 是做什么的
2. 每个字段怎么填
3. 不同场景下应该怎么配置
4. 哪些内容不应该在这里维护

---

## 1. 先理解：什么是 AI 工具技能

`AI 工具技能` 不是 AI 座席人格，也不是路由规则。  
它的作用只有一个：

**定义 AI 在处理客户消息时，可以调用哪些工具能力。**

比如：

- 查询物流轨迹
- 查询订单
- 检索知识库
- 识别客户发来的图片

可以把它理解成：

**AI 的“可调用工具目录”**

而不是：

- AI 座席本身
- AI 运行策略本身
- 路由规则本身
- 知识库内容本身

### 1.1 它和其他模块的关系

系统里这几个模块的边界要分清：

- `AI 座席`
  - 定义 AI 是谁
  - 维护人格设定、服务场景、系统提示词、说明

- `AI 运行策略`
  - 定义 AI 遇到不同情况先做什么
  - 例如：客户问物流先查物流、客户发图片先走图片识别

- `AI 工具技能`
  - 定义“查物流”这个工具怎么接
  - 定义“图片识别”这个工具怎么跑

- `知识库`
  - 维护知识内容本身
  - `知识库检索` 工具只是调用它

一句话：

- `AI 座席` = 角色
- `AI 运行策略` = 什么时候调用
- `AI 工具技能` = 调什么、怎么调

---

## 2. 什么时候需要新建 AI 工具技能

只有在下面这些情况，才需要新建：

### 2.1 新增一个 AI 可调用工具

例如：

- 新增“物流轨迹查询”
- 新增“图片识别”
- 新增“订单查询”

### 2.2 同类工具，但接了不同系统

例如：

- 同样是“订单查询”，但接的是另一个 ERP / OMS
- 同样是“物流查询”，但接的是另一家物流商接口

### 2.3 现有工具需要不同配置

例如：

- 一个图片识别工具用 Gemini
- 另一个图片识别工具用 OpenAI 兼容模型

---

## 3. 什么情况不应该在这里维护

下面这些内容不应该在 `AI 工具技能` 里维护：

### 3.1 AI 的人格、说话方式、服务场景

这些在：

- `AI 座席`

### 3.2 回复前要不要先查什么

这些在：

- `AI 运行策略`

### 3.3 知识内容本身

这些在：

- `知识库`

### 3.4 人工/AI 路由规则

这些在：

- 路由规则
- SLA
- 事项 / Case 状态流转

---

## 4. 页面结构总览

当前 `AI 工具技能` 维护页分 4 块：

1. `基础信息`
2. `工具执行`
3. `补充资料`
4. `高级设置`

实际维护时：

- 日常主要填写 `基础信息`
- 真正接工具时主要填写 `工具执行`
- `补充资料` 用来给 AI 提供参考说明
- `高级设置` 通常不需要改

如果是第一次上手，建议原则：

- 先把 `基础信息` 和 `工具执行` 配好
- 其他先不要动

## 5. 基础信息怎么填写

### 5.1 工具技能名称

字段：

- `工具技能名称`

作用：

- 给管理员和运营看
- 用于区分不同工具

怎么填：

- 用业务名称
- 让人一看就知道这个工具干什么

推荐写法：

- `物流轨迹查询`
- `订单查询`
- `消息图片识别`
- `知识库检索`

不要这样写：

- `工具1`
- `test`
- `new-skill`

---

### 5.2 标识

字段：

- `标识`

作用：

- 系统内部识别这个工具技能的唯一标识

怎么填：

- 一般可以留空
- 系统会根据名称自动生成

建议：

- 除非你明确要稳定命名，否则不要手动改

示例：

- `track_shipment_helper`
- `message_image_recognition`

---

### 5.3 用途说明

字段：

- `用途说明`

作用：

- 给人看，不是给模型执行的主说明
- 帮助团队快速理解这个工具是做什么的

怎么填：

- 一句话到三句话
- 说清楚：什么时候会用到它

示例：

- `当客户询问物流状态时，调用外部物流系统查询轨迹。`
- `当客户发来商品、运单或截图图片时，识别图片中的文字和关键信息。`
- `当 AI 需要从知识库里找售后政策、规则或常见问题答案时，调用此工具。`

---

### 5.4 状态

字段：

- `状态`

选项：

- `草稿`
- `启用`
- `停用`

怎么用：

- `草稿`
  - 还在配置中
  - 不建议让 AI 正式使用

- `启用`
  - 已可投入使用

- `停用`
  - 保留配置，但不继续使用

建议：

- 没调通前先用 `草稿`
- 联调完再切 `启用`

---

### 5.5 给 AI 的操作说明

字段：

- `给 AI 的操作说明`

这是最重要的字段之一。

作用：

- 告诉 AI：
  - 什么时候调用这个工具
  - 需要什么输入
  - 返回内容怎么理解
  - 失败了怎么办

推荐结构：

```md
# 这个工具做什么

# 什么时候使用

# 需要什么输入

# 返回什么结果

# 失败时怎么处理
```

建议写法：

### 物流轨迹查询示例

```md
# 这个工具做什么
查询运单的最新物流轨迹和节点信息。

# 什么时候使用
当客户询问包裹在哪里、什么时候派送、为什么还没到时使用。

# 需要什么输入
运单号 trackingNumber。

# 返回什么结果
返回当前物流状态、最近扫描节点、扫描时间、下一步节点信息。

# 失败时怎么处理
如果没有运单号，先向客户澄清；如果接口返回空结果，提示客户核对运单号或转人工。
```

### 图片识别示例

```md
# 这个工具做什么
识别客户发送图片中的文字、截图内容或关键字段。

# 什么时候使用
当客户发送截图、单据、运单图、付款截图、商品图片并希望 AI 理解内容时使用。

# 需要什么输入
图片 URL 或系统可访问的图片地址。

# 返回什么结果
返回图片中的主要文字、关键信息和识别摘要。

# 失败时怎么处理
如果图片无法读取，提示客户重新发送更清晰的图片；必要时转人工。
```

## 6. 工具执行怎么填写

这是决定“工具实际如何运行”的区域。

一个工具技能里可以维护多个工具，但日常建议：

- 一个技能先只维护一个主工具

这样最清楚，也最容易排查。

---

### 6.1 工具类型

字段：

- `工具类型`

当前常见类型：

- `物流查询`
- `订单查询`
- `知识库检索`
- `图片识别`
- `自定义工具`

怎么选：

#### 物流查询

用于：

- 运单轨迹
- 物流节点
- 签收状态

#### 订单查询

用于：

- 查订单状态
- 查金额
- 查售后单

#### 知识库检索

用于：

- 从知识库模块中检索文档内容

注意：

- 这不是录入知识内容
- 知识内容在 `知识库` 模块维护

#### 图片识别

用于：

- 识别客户发来的图片
- OCR 提取
- 截图理解

#### 自定义工具

用于：

- 系统内置类型不够时
- 用你们自己的工具键接入

---

### 6.2 自定义工具键

字段：

- `自定义工具键`

只有在工具类型选择：

- `自定义工具`

时才需要填。

作用：

- 指向后端已经支持的自定义执行器/能力键

什么时候填：

- 只有你们后端已经明确支持一个自定义工具时才填

示例：

- `recognize_message_image`
- `create_ticket`
- `extract_receipt_fields`

如果后端没有对应实现，不要乱填。

---

### 6.3 启用

字段：

- `启用`

作用：

- 控制这个工具绑定是否生效

建议：

- 联调前可先关闭
- 联调通过再打开

---

### 6.4 使用大模型

字段：

- `使用大模型`

作用：

- 表示这个工具运行时，是否要调用模型来做理解/识别/解析

什么时候开：

- 图片识别
- 文本提取
- 复杂分类
- OCR 场景

什么时候关：

- 纯 API 查询
- 纯数据库查询
- 纯知识库检索

举例：

#### 物流查询

- 通常 `关`

原因：

- 物流接口本身返回结构化数据
- 不需要模型先理解

#### 图片识别

- 通常 `开`

原因：

- 需要模型理解图片内容

---

### 6.5 模型配置

字段：

- `模型配置`

前提：

- 必须先打开 `使用大模型`

作用：

- 指定这个工具运行时使用哪一个模型配置
- 这里选的是模型资源池里已有的配置

注意区分：

- `AI 座席模型`
  - 用于和客户对话

- `工具模型`
  - 用于执行这个工具

例如：

- AI 座席回复用 OpenAI
- 图片识别工具用 Gemini

这是允许且合理的。

---

### 6.6 接口地址

字段：

- `接口地址`

什么时候需要：

- `物流查询`
- `订单查询`
- 其他外部 API 类工具

作用：

- 填外部系统的基础域名

示例：

- `https://openapi.example.com`
- `https://api.erp.xxx.com`

不要填：

- 只填一个路径 `/track`
- 带一堆临时 query 参数的测试 URL

一般建议：

- 只填基础地址
- 具体接口路径放到 `接口路径`

---

### 6.7 请求方式

字段：

- `请求方式`

常见值：

- `GET`
- `POST`
- `PUT`

怎么填：

- 按对方接口文档来

例如：

- 查物流常见 `GET` 或 `POST`
- 提交动作常见 `POST`

---

### 6.8 接口路径

字段：

- `接口路径`

作用：

- 填具体 API 路径模板

示例：

- `/orders/{orderId}`
- `/track?waybill={trackingNumber}`
- `/webopenplatformapi/api/logistics/trace`

这里应该填什么，要看对方接口文档。

---

### 6.9 Token / API Key

字段：

- `Token / API Key`

作用：

- 用于调用外部 API

什么时候需要：

- 外部系统有 key/token 鉴权时

注意：

- 这是敏感信息
- 录入时要确保来自正式授权

---

### 6.10 鉴权方式

字段：

- `鉴权方式`

当前常见选项：

- `none`
- `bearer`
- `header`
- `hmac_sha256`

怎么理解：

#### none

- 不做鉴权

#### bearer

- 常见 `Authorization: Bearer xxx`

#### header

- 用自定义 Header 放 token
- 例如 `X-Api-Key: xxx`

#### hmac_sha256

- 需要按接口规则生成签名

什么时候用：

- 对方文档明确要求签名

---

### 6.11 鉴权 Header 名称

字段：

- `鉴权 Header 名称`

作用：

- 定义 token 或签名放在哪个 Header 里

常见值：

- `Authorization`
- `X-Api-Key`
- `X-Access-Token`

什么时候填：

- `bearer`
- `header`
- 某些签名接口

---

### 6.12 签名密钥（可选）

字段：

- `签名密钥（可选）`

作用：

- 给签名鉴权用

什么时候填：

- 只有对方接口需要签名算法时

比如某些 ERP / 物流商接口会要求：

- 时间戳
- appId
- secret
- digest / 签名

这类场景才需要。

如果只是普通 token，不需要填。

---

### 6.13 超时（毫秒）

字段：

- `超时（毫秒）`

作用：

- 控制外部接口等待多久

建议值：

- 普通查询：`3000 ~ 5000`
- 较慢接口：`5000 ~ 10000`

不要设太大：

- 否则 AI 会卡住太久

---

### 6.14 高级配置（可选）

字段：

- `高级配置（可选）`

作用：

- 放一些额外的 JSON 配置

什么时候用：

- 只有当前表单字段不够表达时

普通维护建议：

- 留空
- 或保留默认 `{}``

除非研发明确告诉你要填什么，否则不要改。

---

## 7. 补充资料怎么填写

这部分是给 AI 提供额外参考信息，不是执行配置。

字段：

- `标题`
- `内容`

适合放什么：

- 外部接口说明摘要
- 字段解释
- 返回码说明
- 特殊业务规则

示例：

### 物流工具补充资料示例

标题：

- `物流状态说明`

内容：

```md
签收状态：
- delivered = 已签收
- in_transit = 运输中
- failed_attempt = 派送失败

如果返回空轨迹，不要直接下结论丢件，先提示客户核对运单号。
```

### 图片识别补充资料示例

标题：

- `图片识别注意事项`

内容：

```md
优先提取：
- 运单号
- 金额
- 日期
- 截图中的错误提示

如果图片模糊，只能返回“识别不清”，不要编造结果。
```

什么时候需要补充资料：

- 工具比较复杂
- 有一些领域知识需要告诉 AI

什么时候不需要：

- 简单查询接口
- 没有额外规则

---

## 8. 高级设置怎么理解

这一块是高级区，通常不需要改。

如果你不是研发或实施，不建议修改。

包含：

- `Trigger Hints JSON`
- `Input Schema JSON`
- `Output Schema JSON`
- `Policy Config JSON`
- `Execution Strategy JSON`
- `签名脚本 / 转换脚本`

### 8.1 Trigger Hints JSON

作用：

- 给系统提示“什么情况下更可能用到这个工具”

如果不懂：

- 不要改

---

### 8.2 Input Schema JSON

作用：

- 定义工具输入结构

例如：

- 需要 `trackingNumber`
- 需要 `imageUrl`

如果不懂：

- 不要改

---

### 8.3 Output Schema JSON

作用：

- 定义工具输出结构

如果不懂：

- 不要改

---

### 8.4 Policy Config JSON

作用：

- 控制一些运行限制和保护策略

如果不懂：

- 不要改

---

### 8.5 Execution Strategy JSON

作用：

- 定义工具执行策略

如果不懂：

- 不要改

---

### 8.6 签名脚本 / 转换脚本

作用：

- 用于复杂签名
- 用于特殊请求转换
- 用于特殊返回结果转换

什么时候需要：

- 对方 API 不只是 token，而是有签名算法
- 需要动态生成 digest、timestamp、header

如果你只是普通维护人员：

- 不要随便写
- 交给研发或实施

---

## 9. 不同场景如何配置

下面给几个实际示例。

---

## 9.1 场景一：物流轨迹查询

适用场景：

- 客户问“我的快递到哪了”
- 客户问“为什么还没到”
- 客户问“什么时候派送”

推荐配置：

### 基础信息

- 工具技能名称：`物流轨迹查询`
- 用途说明：`当客户询问物流状态时，调用物流接口查询轨迹。`
- 状态：`启用`

### 工具执行

- 工具类型：`物流查询`
- 启用：`开`
- 使用大模型：`关`
- 接口地址：填物流商基础地址
- 请求方式：按接口文档
- 接口路径：按接口文档
- Token / API Key：按接口文档
- 鉴权方式：按接口文档
- 超时：`5000`

### 补充资料

- 可放物流状态码说明

---

## 9.2 场景二：图片识别

适用场景：

- 客户发截图
- 客户发运单图
- 客户发付款截图
- 客户发商品图片，希望 AI 理解内容

推荐配置：

### 基础信息

- 工具技能名称：`消息图片识别`
- 用途说明：`识别客户发送图片中的文字和关键信息。`
- 状态：`启用`

### 工具执行

- 工具类型：`图片识别`
- 启用：`开`
- 使用大模型：`开`
- 模型配置：选择合适的图片理解模型

通常这类工具：

- 不一定需要外部 API 地址
- 关键是模型配置

### 补充资料

- 可以写“优先识别运单号、金额、日期、错误提示”

---

## 9.3 场景三：知识库检索

适用场景：

- 客户问退款政策
- 客户问售后规则
- 客户问 FAQ

推荐配置：

### 基础信息

- 工具技能名称：`知识库检索`
- 用途说明：`从知识库模块中检索和当前问题相关的文档内容。`
- 状态：`启用`

### 工具执行

- 工具类型：`知识库检索`
- 启用：`开`
- 使用大模型：通常 `关`

注意：

- 这里不维护知识内容
- 知识内容要去 `知识库` 模块里维护

---

## 9.4 场景四：订单查询

适用场景：

- 客户问订单状态
- 客户问是否已付款
- 客户问退款单状态

推荐配置：

### 基础信息

- 工具技能名称：`订单查询`
- 用途说明：`调用 ERP/OMS 查询订单状态与关键字段。`

### 工具执行

- 工具类型：`订单查询`
- 使用大模型：通常 `关`
- 外部接口字段按 ERP 文档填写

## 10. 常见错误

### 10.1 把 AI 座席人格写进工具技能

错误：

- 在这里写“你是一个温柔客服”

正确：

- 这应该写到 `AI 座席`

---

### 10.2 把“什么时候调用”写成路由规则

错误：

- 直接在工具里试图控制所有流程

正确：

- 工具只定义能力
- 什么时候调用由 `AI 运行策略` 决定

---

### 10.3 把知识内容维护在工具技能里

错误：

- 在补充资料里堆大量 FAQ 正文

正确：

- 正式知识内容放到 `知识库`
- 工具技能这里只保留少量工具级说明

---

### 10.4 不理解就乱改高级设置

错误：

- 不知道含义却改 JSON

正确：

- 不懂就不要改高级区

---

## 11. 推荐维护流程

建议按下面步骤做：

1. 先明确这是不是一个“工具能力”
2. 写好名称、用途说明、给 AI 的操作说明
3. 选对工具类型
4. 如果是外部 API，先确认接口文档
5. 如果要用模型，再选择模型配置
6. 补少量必要资料
7. 先保存为 `草稿`
8. 联调通过后再切 `启用`

---

## 12. 一句话记忆

维护 `AI 工具技能` 时，只记住这句话：

**这里维护的是“AI 会调用什么工具，以及这个工具怎么跑”。**

不是维护：

- AI 人格
- 路由规则
- SLA
- 知识正文

如果团队需要，我建议后续再补两份配套文档：

1. `AI 运行策略怎么维护`
2. `AI 座席怎么维护`

这样三块就完整了。
